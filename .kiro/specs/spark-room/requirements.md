# Spark Room 要件

## 概要

**Spark Room** は、人間とAIエージェントが同じ共有ルームに参加し、アイデアを検討してPRD（製品要求仕様書）を完成させる、ローカルで動作するマルチプレイヤー・ワークスペースである。

人間はブラウザーから参加する。

AIエージェントは、KiroがMCP（Model Context Protocol）ツールを使って「Purin」として参加する。

人間とPurinは、同じボード、エージェントタスク、活動ログ、PRDを読み書きする。

このMVPでは、別途AI APIキーを設定しない。

Kiroのセッションで動くAIエージェントが、MCP経由で共有ルームのチームメイトになる。

## 用語

**人間メンバー**：ブラウザーからルームへ参加し、アイデアを出し、提案と成果物を最終判断する参加者。

**AIエージェント**：MCPツールを使ってルームの状態を読み、担当タスクを実行し、提案やPRD草案を作るKiroエージェント。

**エージェントタスク**：人間がAIエージェントへ割り当てる、質問、提案、レビュー、PRD草案作成などの作業依頼。

**提案**：AIエージェントが作成した、承認されるまで共有ボードへ反映されないカードまたはPRD草案。

## 対象としないこと

- Kiroセッションをブラウザーのボタンだけで自動起動すること
- インターネット公開、ユーザー認証、組織管理
- 外部の生成AI APIをアプリケーションから直接呼び出すこと
- 人間の最終判断なしにAIエージェントが決定事項や完成PRDを確定すること
- 永続的に自律実行するバックグラウンドエージェント

## 要件

### 要件 1：共有ルームの作成と人間メンバーの参加

**ユーザーストーリー**：人間メンバーとして、チームで使う共有ルームを作成し、ほかの人間メンバーを招待したい。

#### 受け入れ基準

1. ユーザーがルーム名と表示名を入力して作成を要求したとき、システムは一意のルームIDを持つ共有ルームを作成しなければならない。
2. ユーザーが有効なルームURLを開いたとき、システムは表示名の入力を求め、その人間メンバーをルームへ参加させなければならない。
3. システムは、ルームURLとルームIDを人間メンバーがコピーできる形で表示しなければならない。
4. 人間メンバーが入室または退出したとき、システムは参加者一覧と活動ログを更新しなければならない。
5. 存在しないルームを開いたとき、システムは状況を説明し、新しいルームの作成画面へ戻る導線を表示しなければならない。

### 要件 2：人間とエージェントの参加者表示

**ユーザーストーリー**：チームメンバーとして、誰がルームにいて、誰が人間かAIエージェントかを確認したい。

#### 受け入れ基準

1. システムは人間メンバーとAIエージェントを同じ参加者一覧に表示しなければならない。
2. システムは各参加者について、表示名、種別、人間またはエージェント、役割、在室状態を表示しなければならない。
3. PurinがMCPの `join_workspace` ツールを実行したとき、システムはPurinを役割付きのAIエージェントとしてルームへ追加しなければならない。
4. AIエージェントが退出したとき、システムは参加者一覧と活動ログにその状態を反映しなければならない。
5. システムは、ブラウザーからKiroを自動起動できないことと、Kiroでルーム参加指示を実行する手順を表示しなければならない。

### 要件 3：共有アイデアボード

**ユーザーストーリー**：チームメンバーとして、アイデアと決定を全員が見える場所で整理したい。

#### 受け入れ基準

1. システムは各ルームに「課題」「対象ユーザー」「アイデア」「決定事項」の4つのボード領域を表示しなければならない。
2. 人間メンバーがカードを追加、編集、削除したとき、システムは同じルームを開いているすべてのブラウザー参加者へ、ページを再読み込みせずに変更を反映しなければならない。
3. AIエージェントが `propose_card` ツールを実行したとき、システムは作成者、エージェントの役割、提案理由を示す保留中の提案カードを表示しなければならない。
4. 人間メンバーが提案を承認したとき、システムはそのカードを対象のボード領域へ反映し、活動ログへ記録しなければならない。
5. 人間メンバーが提案を却下したとき、システムは提案をボードへ反映せず、却下した事実を活動ログへ記録しなければならない。

### 要件 4：エージェントタスクの割り当てと実行

**ユーザーストーリー**：人間メンバーとして、会議に必要な作業をAIエージェントへ依頼し、その進捗を共有したい。

#### 受け入れ基準

1. 人間メンバーは、質問、提案、レビュー、PRD草案作成のいずれかを種別とするエージェントタスクを作成できなければならない。
2. システムは、各エージェントタスクについて、依頼内容、担当エージェント、状態、作成者、作成日時を表示しなければならない。
3. AIエージェントが `list_agent_tasks` と `update_agent_task` を実行したとき、システムはタスク状態を「未着手」「進行中」「完了」のいずれかとして更新しなければならない。
4. システムは、Kiroへ貼り付けるためのルームIDと未完了タスクを含む参加依頼文をコピーできるようにしなければならない。
5. AIエージェントがタスクを完了したとき、システムは結果へのリンクまたは説明を活動ログへ記録しなければならない。

### 要件 5：AIエージェントの共同作業

**ユーザーストーリー**：チームメンバーとして、AIエージェントが単なる会話相手ではなく、担当作業を持つチームメイトとして振る舞うことを期待する。

#### 受け入れ基準

1. AIエージェントは書き込み前に `read_workspace` を使い、ルームの課題、決定事項、未完了タスクを確認しなければならない。
2. AIエージェントは `post_agent_message` を使い、提案または分析の意図と根拠をチームへ説明しなければならない。
3. AIエージェントは、決定事項のカードを直接確定してはならず、`propose_card` による保留中の提案として追加しなければならない。
4. AIエージェントは、1つの提案タスクで最大3件のカードだけを提案しなければならない。
5. AIエージェントは、人間メンバーの表示名や発言者種別を偽装してはならない。
6. AIエージェントは、割り当てられたタスクの完了後に結果を記録し、次に人間が判断すべき項目を示さなければならない。

### 要件 6：共有状態の同期と記録

**ユーザーストーリー**：チームメンバーとして、ほかの人間やAIエージェントの作業をすぐに確認し、議論の経緯をたどりたい。

#### 受け入れ基準

1. システムは、すべてのルーム変更を共有サーバーへ保存しなければならない。
2. システムは、共有サーバー上で発生したカード、タスク、メッセージ、PRDの変更を、接続中のブラウザーへリアルタイムに配信しなければならない。
3. AIエージェントがMCP経由で書き込んだ変更は、人間メンバーのブラウザー画面へページ再読み込みなしで表示されなければならない。
4. システムは、カード追加、提案、承認、却下、タスク状態変更、PRD草案保存を、操作者と時刻を含む活動ログへ記録しなければならない。
5. 人間メンバーがルームを再読み込みしたとき、システムは保存済みのルーム状態を復元しなければならない。

### 要件 7：PRD草案の作成、承認、出力

**ユーザーストーリー**：チームメンバーとして、議論の結果を次の開発作業へ使えるPRDとして持ち帰りたい。

#### 受け入れ基準

1. 人間メンバーはPRD草案作成タスクをAIエージェントへ割り当てられなければならない。
2. AIエージェントが `save_prd_draft` を実行したとき、システムはMarkdown形式のPRD草案を保存し、草案であることと作成者を表示しなければならない。
3. 生成するPRDには、少なくとも「背景と課題」「対象ユーザー」「目的」「主要機能」「対象外」「成功指標」「未決定事項」「次の作業」を含めなければならない。
4. 人間メンバーはPRD草案を編集、承認、または差し戻しできなければならない。
5. システムは承認済みPRDをコピーまたはMarkdown形式でダウンロードできるようにしなければならない。
6. AIエージェントは、PRDを承認済みとして確定してはならない。

### 要件 8：ローカルMVPとしての利用しやすさ

**ユーザーストーリー**：初めて使う人間メンバーとして、ローカルの共有サーバーとKiroを使って、短時間で共同作業を始めたい。

#### 受け入れ基準

1. システムはログインと外部AI APIキーなしで、ローカルサーバーの起動、ルーム作成、人間参加までを行えるようにしなければならない。
2. システムは、Kiroへ接続するMCPサーバーの設定手順と、AIエージェントを招待する手順を日本語で示さなければならない。
3. システムは、共有サーバーが起動していない場合、問題と復旧手順を表示しなければならない。
4. システムはキーボードで主要な操作を開始でき、入力欄と操作ボタンに識別可能なラベルを提供しなければならない。
5. システムは、ローカルMVPのデータが開発用の共有サーバーへ保存され、認証なしで動作することを表示しなければならない。

## デモの完了条件

次の流れを実演できること。

1. 人間メンバーAが「ひとり暮らしの学生の食材ロスを減らすアプリ」のルームを作成する。
2. 人間メンバーBが同じルームURLから参加し、ボードに対象ユーザーのカードを追加する。
3. Aが「学生向けの機能を3件提案して」というエージェントタスクを作成する。
4. KiroでPurinへルームIDと担当タスクを渡す。
5. PurinがMCPでルームへ参加し、状態を読んで、根拠を説明するメッセージと保留中の提案カードを追加する。
6. AまたはBが提案を承認または却下し、決定事項を更新する。
7. AがPRD草案作成タスクを作成し、PurinがMCP経由でMarkdownのPRD草案を保存する。
8. 人間メンバーがPRDを編集、承認、出力する。
9. 活動ログに人間とPurinの両方の操作が残る。
